Object Detail

Playback Engine

The Playback Engine is the runtime object that resolves transport authority and turns project/workspace state into actual playback and cursor time. It lives on the runtime side of Chord Lab and sits between model data, transport inputs, preview layers, and the timeline-facing views. This object matters because it is where host-vs-internal authority, preview isolation, and arrangement-following behavior either become trustworthy or start to feel brittle.

Definition

The Playback Engine is the runtime object that consumes transport inputs, arrangement state, and preview state to produce audible playback and authoritative cursor time. It resolves whether host or internal transport currently owns the timeline, applies arrangement-following playback rules, and layers scoped preview behavior without turning preview into a second transport. When Chord Lab starts or stops playback, follows the host, runs internal audition, or updates local preview rendering, the Playback Engine is one of the main objects being exercised.

Runtime authority resolver Audio/cursor source Preview-layer host Transport-facing core
Key distinction:
The Playback Engine is not the same as transport authority itself. Transport authority is a contract; the Playback Engine is the runtime object that enforces that contract and feeds the rest of the app.

What This Object Is Not

Is This Object

The runtime engine that decides which timeline is active and renders playback under that decision.

Not This Object

The editor, the transport buttons, or the preview policy itself. Those are consumers or related contracts around the engine.

Boundary Case

Pattern Generator design-mode looping may feel like it owns playback, but it still belongs to preview/session policy layered through the Playback Engine under transport authority.

Core Responsibilities

Responsibility Meaning What It Owns What It Does Not Own Notes
Transport resolution Determines which playback timeline is currently authoritative Resolved runtime playback mode, active playback position, and authority-dependent status High-level product policy itself The rule is "host always wins"; the Playback Engine is where that becomes real behavior.
Arrangement-following render Turns progression, Beat Lane, and performance state into actual playback events Runtime render state and playback caches The source model objects themselves This is why stale playback behavior often points back to engine update/invalidation contracts.
Preview-layer isolation Allows local preview behavior to be heard without becoming a second transport or corrupting main playback Scoped preview rendering and preview-only state handling Tool-local session ownership or editor layout state This boundary has been a repeated source of Pattern Generator bugs and hardening work.

Mutation Touch Points

  • Transport mutations directly affect the Playback Engine by changing which time source is active and whether playback is running.
  • Beat Lane and Generated Pattern mutations affect the engine indirectly by changing the arrangement or beat content it must render.
  • Session mutations affect the engine when they change preview policy, placed-beat context, or local audition intent for the Pattern Generator.

Design Tensions / Failure Modes

  • If preview updates are routed through the same heavy or authority-owning path as main playback, the engine can retrigger notes, feel laggy, or appear to glitch.
  • If host-authority resolution is correct in the engine but not reflected immediately in the UI, users still experience playback as untrustworthy.
  • Because the engine sits at the intersection of transport, arrangement, and preview, weak invalidation boundaries here can make many unrelated workflows feel brittle at once.
