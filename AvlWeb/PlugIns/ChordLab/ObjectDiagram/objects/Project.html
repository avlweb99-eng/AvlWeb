Object Detail

Project

The Project is the top-level persistent document in Chord Lab. It is the root object that contains workspaces and the shared structure needed to save, load, and publish the app's musical state. This object matters because every major model object ultimately lives inside it or is derived from it.

Definition

The Project is the outermost model object that owns the full Chord Lab document. It contains the set of workspaces and any project-level metadata or state that must survive persistence. When we talk about "saving the song" or "publishing the current app state," this is the object at the root of that operation.

Top-level model Persistent root Owns workspaces Document scope
Key distinction:
The Project is not the same as a Workspace. A workspace is one contained musical working area; the project is the container for all of them.

What This Object Is Not

Is This Object

The thing that contains the workspace list and acts as the root of serialization and project-wide mutation snapshots.

Not This Object

A single workspace, progression, or Beat Lane. Those are nested inside the project rather than being the root document.

Boundary Case

Some "project-wide" UI may really just be showing the currently selected workspace. The project object itself still sits one level above that working scope.

Relationships In Chord Lab

Related Object Relationship Direction Notes
Workspace Contains one or more workspace objects Downstream This is the most important structural relationship for the project object.
Editor Editor presents and manipulates a selected workspace from the project Downstream consumer The editor rarely works against the whole project at once, but it still depends on the project root.
Playback Engine Consumes project-derived snapshots for playback truth Downstream consumer Playback may use a project snapshot even when the user is focused on only one workspace.

Mutation Touch Points

  • Workspace-level value mutations ultimately write into project-contained state.
  • Project-wide publish and undo behavior commonly snapshot from the project root.
  • Selection mutations that change active workspace are project-scoped because they retarget which child workspace the editor is using.

Design Tensions / Failure Modes

  • When the project root is used as the default refresh or undo boundary for every small change, cheap local interactions can become much more expensive than they need to be.
  • It is easy for UI and tool code to blur project scope with workspace scope, which makes ownership and update contracts harder to reason about.
  • Pattern Generator and Beat Lane work highlight this tension because many edits are workspace-local but can still accidentally travel through project-wide paths.
