Object Detail

Progression

The Progression is the workspace-owned arrangement of chord blocks over musical time. It holds the harmonic timeline the user sees in the Chord Roll and that playback follows for chord-driven behavior. This object matters because it is the main source of truth for harmony, chord timing, and progression-level musical structure inside a workspace.

Definition

The Progression is the model object that represents the chord sequence and chord timing for a workspace. It is the harmonic timeline that the Chord Roll visualizes and that many playback, suggestion, and editing behaviors derive from. When the user adds, removes, moves, or edits chords, they are usually mutating the progression or objects it owns.

Harmony timeline Workspace-owned Arrangement object Playback-relevant
Key distinction:
The Progression is not the same as a single Chord Block. A Chord Block is one placed harmony event; the Progression is the object that contains and orders those events.

What This Object Is Not

Is This Object

The thing that owns chord blocks, their timing, and the harmonic arrangement the user edits in the Chord Roll.

Not This Object

A workspace, which owns the progression, or a Beat Lane, which owns placed beat arrangement instead of harmony arrangement.

Boundary Case

Loop-related UI often feels transport-like, but progression-level loop settings still belong to the arrangement/harmonic scope rather than to transport authority itself.

Progression Diagram

The Progression sits under Workspace and above the individual chord blocks, while interacting closely with playback, suggestions, and editor views.

Mutation Touch Points

  • Most chord-related value mutations write into the progression or directly into chord blocks it owns.
  • Selection mutations in the Chord Roll usually target progression-owned chord blocks.
  • Transport and playback consumers rely on progression truth, but transport authority itself remains a separate contract above it.

Design Tensions / Failure Modes

  • It is easy for progression edits to trigger broader-than-necessary UI churn if the editor treats any change in progression state as a reason to refresh unrelated domains.
  • Some progression-facing controls can blur the line between arrangement truth and transport truth, especially around loop behavior.
  • Because progression sits at the center of harmony, suggestions, playback, and selection, weak update contracts here can make many parts of the app feel stale or fragile at once.
