Object Detail

Chord Block

A Chord Block is one placed harmony event inside a progression. It carries the musical and timing information for a single chord occurrence that the user can see, select, move, and edit in the Chord Roll. This object matters because many of the most frequent mutations in Chord Lab ultimately target one Chord Block at a time.

Definition

A Chord Block is the atomic harmony object in the progression timeline. It represents one placed chord event with its own harmony, timing, and performance-related properties. When a user clicks a chord, changes its quality, moves it in time, or edits its duration, they are usually operating on one or more chord blocks.

Atomic harmony event Progression-owned Highly editable Selection hotspot
Key distinction:
A Chord Block is not the whole progression. It is one member of the progression’s chord arrangement, with its own local properties and timing.

What This Object Is Not

Is This Object

A single chord event such as “bar 2, beat 1, C minor 7 for two beats” placed on the progression timeline.

Not This Object

The progression itself, or a generic UI selection highlight. The Chord Block is the model object underneath that highlight.

Boundary Case

Performance settings can feel external to the block, but in selected-chord scope they may behave like block-local properties or block-targeted overrides.

Chord Block Diagram

A Chord Block lives inside the progression and is consumed heavily by inspector, playback, suggestion, and selection systems.

Relationships In Chord Lab

Related Object Relationship Direction Notes
Progression Chord Block is contained and ordered by the progression Upstream The progression is the arrangement container; the block is one member of it.
Chord Roll View Shows and manipulates block placement visually Downstream consumer Most direct manipulation of block timing and selection happens here.
Chord Inspector Reads and edits block-local properties Downstream consumer The inspector is one of the most common write paths into block state.
Performance Inspector May target the selected block when using selected-chord scope Downstream consumer / overlay relationship This is where block-local vs broader-scope performance behavior can get subtle.
Suggestion systems Use block selection and harmonic context to generate relevant options Downstream consumer Selections frequently retarget suggestion context by retargeting the current chord block.

Mutation Touch Points

  • Selection mutations frequently target one or more chord blocks in the Chord Roll.
  • Value mutations such as root, quality, duration, start beat, and velocity edits often write directly into block-local state.
  • Performance-related changes can behave like block-targeted mutations when the current scope is “selected chord.”

Design Tensions / Failure Modes

  • Because Chord Blocks are such common targets, weak selection or value-update contracts here can make the entire editor feel laggy or stale very quickly.
  • Block-local behavior can get blurred with broader workspace or performance-scope behavior, especially when selected-chord and all-chords modes are mixed in the same UI.
  • If editor refreshes treat every block change as a reason to rebuild unrelated domains, ordinary chord editing becomes much more expensive than it needs to be.
