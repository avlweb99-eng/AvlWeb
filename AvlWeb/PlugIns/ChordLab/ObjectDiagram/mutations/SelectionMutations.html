Mutation Detail

Selection Mutations

Selection mutations are state changes that alter the current edit target or focused object. They are common, frequent, and should be cheap. Their main job is to retarget the UI and tool context, not to rebuild unrelated model or playback state.

Back to Object Diagram Hub Related: Value Mutations Related: Session Mutations Related: Update Contracts

Definition

A selection mutation happens when the app changes what subsequent edits apply to. This may be a global selection in the main editor or a local selection inside a tool session.

Frequent UI-targeted Usually local Should be lightweight
Key distinction:
A selection mutation changes the current target. It does not necessarily change the underlying musical data.

What This Is Not

Is Selection

Clicking a chord block, selecting multiple chord blocks, selecting a lit hit in the Pattern Generator.

Not Selection

Pressing Play, Undo, opening a menu, changing tempo, resizing a window.

Can Be Both

Double-clicking a Beat Lane block may both select the block and open a new session context in the Pattern Generator.

Selection Mutation Diagram

Selection mutations flow mainly into inspector state, focused tool context, and visual selection surfaces. They should not behave like a full model mutation.

Examples

Global Selection Mutation

Click a chord block in the Chord Roll. The active editing target becomes that chord block.

Local Session Selection Mutation

Click a lit step in the Pattern Generator. The active editing target becomes that step inside the session.

Selection + Session Mutation

Double-click a Beat Lane block to open the Pattern Generator in placed-beat mode. This changes session context and may also define a focused object.

Non-Examples

  • Pressing Play or Stop
  • Changing tempo or loop length
  • Toggling Auto-audition
  • Opening a menu without changing edit target
  • Dragging a resize handle
  • Toggling a step on or off in the Pattern Generator This is a generated pattern mutation, not a selection mutation.

Selection Mutation Contract

Mutation Authoritative Change Must Update Must Not Update Notes
Select chord block The current chord edit target changes in editor or processor-facing selection state. Chord inspector, performance inspector, suggestion context, chord selection visuals Beat Lane model, generated pattern library, arrangement playback caches, broad layout state One of the most common editor interactions, so it must stay cheap.
Select workspace The active workspace scope changes and later edits target a different workspace. Workspace-scoped controls, visible summaries, any selection-dependent context that is workspace-specific Unrelated layout state, preview playback logic that does not depend on workspace context, generic full refresh This is selection-like because it changes what subsequent edits apply to.
Select Pattern Generator step The local hit target changes inside the active Pattern Generator session. Step inspector, selected-step visuals, local session summaries Beat Lane model, generated pattern asset contents, unrelated editor panels, global transport state This must stay local to the dialog or tool session.

Selection Mutation Types In Chord Lab

Mutation Category What Changes Notes
Select chord block Global selection Primary chord edit target Should refresh inspectors and suggestion context only.
Marquee select multiple chord blocks Global selection Primary + secondary selected blocks Still a selection mutation even though multiple blocks are involved.
Select workspace Global selection / scope Current workspace context This is selection-like because subsequent edits target a different workspace.
Select Beat Lane block Global selection or session seed Current Beat Lane target Whether this is explicit today depends on UI behavior, but the contract should be clear.
Select step in Pattern Generator Local session selection Current hit being edited Should stay local to the Pattern Generator session.
Switch performance scope Selection-like scope mutation Which target performance edits apply to This should not behave like a full value mutation by itself.

Design Tensions / Failure Modes

  • Some UI interactions mix selection change with session change, especially around opening the Pattern Generator from Beat Lane.
  • The editor has historically treated many processor changes as broad refresh events, which can make selection look more expensive than it should be.
  • Selection in the main editor and selection inside a local tool session need to stay distinct.
  • If clicking a new target causes broad relayout, playback glitches, or later “repair refreshes” are needed to make the UI look correct, the selection contract is too broad.

Open Questions

  • Should Beat Lane block click establish a formal global selection target, or only a local interaction target?
  • Should opening the Pattern Generator on a placed beat explicitly create both a session mutation and a selection mutation?
  • Which selection changes should be represented as processor state versus local dialog/session state?
