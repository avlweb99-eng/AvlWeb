Session Mutations

Session mutations are state changes that alter tool context, working mode, or local editing scope. They change how Chord Lab is currently operating without necessarily changing the underlying musical data. This distinction matters because session changes should retarget behavior, visibility, and local playback logic without pretending to be model edits.

Definition

A session mutation happens when Chord Lab changes the current working context of a tool or surface. This includes opening or closing the Pattern Generator, switching it between design mode and placed-beat mode, or changing editor-local context such as the active inspector tab. The core question on this page is: when context changes, what should be re-targeted locally and what should stay out of the mutation?

Context change Mode-sensitive Often local Should stay scoped
Key distinction:
A session mutation changes how a tool is operating. It is different from a value mutation, which changes stored data, and different from a selection mutation, which changes the current target inside an existing context.

What This Is Not

Is Session Mutation

Opening Pattern Generator in standalone design mode or switching it to a placed-beat editing context.

Not Session Mutation

Changing a step velocity or toggling a step on/off. Those are generated pattern value mutations.

Boundary Case

Double-clicking a Beat Lane block can trigger both a session mutation and a selection-related consequence, but the session aspect is the tool opening and context switch.

Session Mutation Diagram

Session mutations start from mode or context changes, rewrite local working state, and then retarget only the views, cursors, and tool behaviors that depend on that session.

Examples

Open Blank Design Session

Open Pattern Generator from an empty Beat Lane area. The tool enters standalone design context even though no pattern data has changed yet.

Open Placed-Beat Session

Double-click a Beat Lane block. The Pattern Generator now works against that placed-beat context and should follow arrangement truth.

Switch Local Editing Context

Change the active inspector tab or explicitly change the Pattern Generator mode. The tool context changes without necessarily touching stored values.

Non-Examples

  • Clicking a lit step so the step inspector targets it This is a local selection mutation.
  • Changing a step’s velocity or accent This is a value mutation on the generated pattern asset.
  • Starting playback This is a transport mutation.
  • Resizing the Pattern Generator dialog This is a layout mutation.

Session Mutation Contract

Mutation Authoritative Change Must Update Must Not Update Notes
Open standalone Pattern Generator Tool-local session mode and local references change to design context Pattern Generator subtitle, empty-state behavior, independent audition policy, visible controls Unrelated project data, Beat Lane model, global selection unless explicitly intended This is the clearest example of session-without-value-change.
Open placed-beat Pattern Generator Tool-local session mode and opened Beat Lane block reference change Pattern Generator mode label, block-local cursor policy, arrangement-following behavior, relevant summaries Unrelated dialogs, generic global relayout, unrelated playback paths This can be paired with a selection consequence, but the session change is still its own mutation.
Switch active tool context Current local mode, tab, or inspector context changes Visible controls, enabled states, local summaries, local preview policy if mode-dependent Stored musical data, unrelated model surfaces, broad project publish Session mutations are often frequent enough that they need to stay light.

Session Mutation Types In Chord Lab

Mutation Category What Changes Notes
Open Pattern Generator in design mode Tool session Standalone pattern-design context becomes active This should create a local editing world without implying a shared model write.
Open Pattern Generator on Beat Lane block Tool session Placed-beat edit context becomes active with block and pattern references This is one of the most important session boundaries in Chord Lab.
Fall back from placed-beat to design mode Tool session Pattern Generator stops claiming placed-block truth and returns to reusable-pattern context Useful when the originally opened context is no longer the honest active context.
Switch active inspector or view tab Editor session Current editing surface or sub-panel focus changes Not every tab change is equally important, but it is still a session-level context change.
Open or close local preview intent Tool session / local behavior Whether the tool is currently configured to preview in a certain local mode changes This becomes especially important when separating preview from transport authority.
Clear or reset session context Tool session Opened references and local mode assumptions are removed Should leave the tool in a valid, explicit fallback state rather than stale mixed state.

Design Tensions / Failure Modes

  • Pattern Generator has both standalone and placed-beat behaviors, so session boundaries must be explicit or the tool feels dishonest.
  • Session mutations can easily get mixed with selection or value mutations, which leads to partial refreshes and “repair refresh” behavior later.
  • If tool-local preview logic is wired through generic processor-wide refresh paths, session changes become more expensive and more fragile than they should be.

Open Questions

  • Which session states should live only in local dialog state, and which should be visible to the processor or playback engine?
  • Should every tool-local mode switch have an explicit update contract enum, or are some changes lightweight enough to stay internal?
  • How should future Beat Lane and loop-region features expose session state so arrangement-following behavior stays understandable?
