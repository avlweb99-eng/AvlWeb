Generated Pattern Mutations

Generated pattern mutations are state changes that create, modify, rename, or delete reusable beat pattern assets. They affect shared pattern data that may be edited in the Pattern Generator and referenced by one or more Beat Lane blocks. This distinction matters because generated pattern edits sit directly at the boundary between local tool behavior, shared assets, and arrangement consequences.

Definition

A generated pattern mutation happens when Chord Lab changes the data or identity of a reusable pattern asset. This includes creating a pattern, renaming it, editing enabled steps or per-step values, and deleting the pattern from the workspace pattern library. The core question on this page is: when a pattern asset changes, what stays local to the Pattern Generator and what must propagate to Beat Lane and playback behavior?

Shared asset change Pattern-library scoped Session-adjacent Arrangement consequences
Key distinction:
A generated pattern mutation changes the pattern asset itself. It is different from a session mutation, which changes tool context, and different from a local step selection mutation, which only changes the currently targeted hit.

What This Is Not

Is Generated Pattern Mutation

Creating a new pattern, toggling a step on/off, changing a step accent, or renaming a saved pattern.

Not Generated Pattern Mutation

Clicking a lit step so the inspector targets it. That is local selection, not a pattern asset change.

Boundary Case

Opening the Pattern Generator on an empty bar is session-level. The mutation becomes a generated pattern mutation only when the new pattern asset is actually created or edited.

Generated Pattern Mutation Diagram

Generated pattern mutations start from pattern editing actions, rewrite shared pattern-library state, and then propagate only to the surfaces and playback paths that depend on that pattern asset.

Examples

Create New Pattern

Start from a blank Pattern Generator session and save the first pattern. A new shared asset is created even though the session itself already existed.

Edit Existing Pattern

Toggle several steps and adjust their velocities. The same pattern asset changes, which means any Beat Lane block referencing it may also need to reflect the update.

Delete Shared Pattern

Delete a generated pattern that is already used in Beat Lane. This is a heavier pattern mutation because it can invalidate placed references.

Non-Examples

  • Clicking a step so the step inspector targets it This is a local session selection mutation.
  • Opening Pattern Generator in design mode This is a session mutation until a pattern asset is actually created or changed.
  • Moving a Beat Lane block that references a pattern This is a Beat Lane mutation, not a generated pattern mutation.
  • Starting or stopping playback These are transport mutations.

Generated Pattern Mutation Contract

Mutation Authoritative Change Must Update Must Not Update Notes
Create pattern A new generated pattern asset is added to the workspace pattern library Pattern lists, Pattern Generator content, any empty-state logic, and local preview tied to the new asset Unrelated Beat Lane blocks, unrelated dialogs, broad global layout Creation is where a blank session becomes backed by shared model state.
Edit pattern contents The events or metadata of an existing generated pattern asset change Pattern Generator grid and inspector, preview behavior, any Beat Lane blocks referencing the asset Unrelated selections, unrelated editor panels, unrelated workspaces This is the most frequent pattern mutation path and needs to stay responsive.
Delete pattern The pattern asset is removed from the workspace library Pattern lists, Pattern Generator context, Beat Lane references, any playback or preview state using the asset Unrelated project data, unrelated transport state, unrelated selections This is typically the heaviest generated pattern mutation because references may need cleanup.

Generated Pattern Mutation Types In Chord Lab

Mutation Category What Changes Notes
Create generated pattern Pattern library mutation A new reusable pattern asset appears in the workspace This is the first shared-model mutation in a blank design session.
Rename generated pattern Pattern metadata mutation Pattern name or descriptive metadata changes Non-musical, but still an authoritative asset change.
Enable or disable step Pattern content mutation Pattern events change, altering what the pattern will play This is easy to confuse with local selection, but it is a true asset edit.
Edit step values Pattern content mutation Velocity, length, accent, or other per-step event data changes These are frequent edits and a major responsiveness hotspot.
Delete generated pattern Pattern library mutation Asset is removed, potentially invalidating Beat Lane placements This is where shared asset lifecycle and arrangement truth collide most directly.
Assign generated pattern to Beat Lane block Cross-boundary mutation A Beat Lane block now references a generated pattern asset The assignment itself is better treated as a Beat Lane mutation, but it creates an important dependency edge for this page.

Design Tensions / Failure Modes

  • Generated pattern edits sit between local tool state and shared arrangement consequences, which makes it easy to over-refresh or under-refresh the system.
  • Pattern Generator responsiveness suffers if every small asset edit behaves like a whole-project publish rather than a scoped pattern-library mutation.
  • Delete pattern behavior is especially sensitive because it can change both the pattern library and Beat Lane truth at the same time.

Open Questions

  • Which generated pattern edits should stay on the lightweight path, and which must intentionally use heavier publish/update behavior because they alter Beat Lane truth?
  • Should the system ever distinguish between local draft edits and committed shared pattern changes, or should asset edits always apply immediately?
  • How should future copy-on-edit or per-placement customization interact with the current shared pattern asset model?
