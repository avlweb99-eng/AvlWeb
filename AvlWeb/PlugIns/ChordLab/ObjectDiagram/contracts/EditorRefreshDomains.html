Editor Refresh Domains

Editor refresh domains define how the main Chord Lab editor consumes invalidations in small, explicit sections instead of rebuilding broadly on every change. They connect mutation/update contracts to the actual UI surfaces that must respond inside the editor. This contract matters because the editor has historically been one of the biggest sources of expensive churn, visible brittleness, and "it only looks right after something else refreshes" behavior.

Definition

An editor refresh domain is a section of the main editor that can update independently in response to a specific invalidation. Instead of treating `refreshFromProcessor()` as a generic "update everything" mechanism, the editor should route only the relevant state changes to the domains that actually depend on them. This contract protects responsiveness, reduces relayout churn, and makes stale UI easier to diagnose.

Scoped UI consumption Domain-based Layout-aware Performance-critical
Key distinction:
Refresh domains are not mutations. They are editor-side consumers of mutation contracts, and their job is to react narrowly instead of broadly.

What This Is Not

Is This Contract

A chord selection change updates only inspector- and suggestion-related domains, while Beat Lane and unrelated workspace controls stay idle.

Not This Contract

"Call `resized()` and `repaint()` at the end of every processor change." That is exactly the broad fallback this contract is trying to replace.

Boundary Case

Some changes really do affect several domains at once, but they should still do so by explicit routing rather than by one generic full-editor refresh.

Editor Refresh Domain Diagram

Mutations emit invalidations, the editor coordinator classifies them, and only the affected domains update or relayout.

Examples

Selection-Only Update

Choosing a different chord block updates the chord inspector, performance scope resolution, and suggestion domains, but does not relayout the whole editor.

Workspace List Update

Changing the actual workspace list rebuilds the workspace selector domain, but ordinary selection changes should not repopulate the combo box each time.

Layout-Only Relayout

Collapsing the hero/header area legitimately changes top-level bounds, so a layout pass is warranted, but it should still be triggered explicitly by layout-affecting state.

Non-Examples

  • "A processor change happened, so rebuild everything" This is the anti-pattern the refresh-domain contract is meant to replace.
  • "Timer updates the playback cursor" That is a lightweight runtime update path, not an editor refresh domain by itself.
  • "Pattern Generator froze while resizing" That is a related failure mode, but it belongs to dialog/layout coupling rather than the main editor domain contract directly.

Editor Refresh Domain Rules

Rule Meaning Applies To Must Prevent Notes
Refresh by domain, not by editor-wide impulse Each invalidation should map to one or more editor domains, not to a generic "refresh everything" action. Main editor coordinator and `refreshFromProcessor()` path Broad churn and accidental coupling This is the core contract behind the current refactor direction.
Layout work is separate from value/selection work Relayout should only happen when structure, visibility, or bounds-affecting state changes. `resized()`, visibility changes, tab changes, hero collapse Unconditional relayout on ordinary data changes One of the most visible wins already made was removing unconditional relayout from the end of refresh.
High-frequency domains must stay cheap Selection, transport status, and common inspector updates should avoid rebuilding unrelated controls. Selection inspector, transport UI, workspace combo, action buttons Lag during ordinary interaction Workspace selector rebuilds and broad repaint tails are classic examples of contract violations.
Cached UI-facing state should gate work The coordinator should compare previous and current UI-relevant state before deciding what to refresh. Editor-side view state cache and refresh router Repeated no-op UI churn This is the practical mechanism that turns the contract into code.

Domain Role Typical Inputs Notes
Project / header domain Workspace summary and high-level controls Workspace selection, workspace list changes, naming, key/scale/tempo/loop values This is where workspace selector churn needs to stay tightly gated.

Transport / action-button domain Playback status and action affordances Undo/redo availability, host/internal status, play-state, selection-dependent actions This is where authority and enabled/disabled states need to be visibly honest.

Design Tensions / Failure Modes

  • The editor file has historically mixed layout, data refresh, and control updates in one broad method, which makes it easy for unrelated work to get dragged into every change.
  • Some operations currently act like "good paths" because they happen to trigger a more complete refresh, which means weaker paths are still under-declaring what should update.
  • If editor domains are not explicit, the refactor can devolve into moving the same generic-refresh behavior into smaller files instead of truly improving responsiveness.

Open Questions

  • Which editor domains should eventually become their own view modules first, once the routing contract is stable?
  • Should the editor coordinator consume shared invalidation enums directly, or translate them into editor-specific domain flags first?
  • How much of the current timer-driven cursor behavior should stay outside normal domain refreshes permanently?
