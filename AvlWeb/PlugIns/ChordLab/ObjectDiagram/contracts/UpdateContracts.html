Update Contracts

Update contracts define what must happen after a mutation changes state. They translate a mutation into explicit invalidations, consumers, and non-consumers so the system becomes correct immediately and predictably. This contract matters because Chord Lab has already shown the cost of weak update rules: stale views, "repair refreshes," heavy generic refreshes, and UI behavior that feels brittle.

Definition

An update contract is the explicit rule set that says what a mutation invalidates, which consumers must refresh, and which parts of the app must stay untouched. In Chord Lab, update contracts sit between mutation types and concrete UI/playback behavior. A good contract keeps correctness deterministic and cost scoped.

Mutation-driven Consumer-aware Scope-limited Correctness-first
Key distinction:
A mutation describes what changed. An update contract describes what the rest of the system must do because that change happened.

What This Is Not

Is Update Contract

"Editing a chord value must refresh the chord inspector, chord roll visuals, and relevant playback caches, but must not trigger a broad editor relayout."

Not Update Contract

"Click the chord inspector slider to change velocity." That is a UI instruction, not the contract that follows from the change.

Boundary Case

"Send a generic change message and let everything refresh" is a fallback mechanism, not a good update contract, even if the code still does it today.

Update Contract Diagram

A mutation changes state, the contract classifies the consequences, and only the declared consumers respond.

Examples

Good Selection Contract

Selecting a chord updates inspectors and selection visuals, but does not rebuild Beat Lane model state or trigger a broad relayout.

Scoped Pattern Contract

Editing a generated pattern step updates the pattern grid, preview, and any Beat Lane blocks that reference that pattern, but does not force a full project publish.

Authority Contract

When the host starts playing, transport UI, playback authority, and cursors switch to host-following immediately while internal controls disable.

Non-Examples

  • "Call refreshFromProcessor() and let everything figure itself out" This is a fallback mechanism, not a clear contract.
  • "The UI looks correct after I move a Beat Lane block" That may reveal a strong contract path, but it is not the contract itself.
  • "The timer repaints every 30 ms" That is runtime behavior, not a mutation-to-consumer rule.

Core Update Contract Rules

Rule Meaning Applies To Must Prevent Notes
Mutations must classify their consequences Every state change should translate into a small explicit invalidation set. All mutation families Guesswork and accidental repair refreshes This is the main design pivot away from generic refresh behavior.
Only declared consumers should react A mutation should refresh only the surfaces that actually depend on the changed state. Editor domains, tool sessions, playback engine Full-window churn and brittle lag This is the contract side of the refresh-domain work already started in the editor.
Non-consumers must stay idle If a surface does not depend on the mutation, it should not update just because a change happened elsewhere. All UI and playback consumers Broad refreshes, repaints, and hidden semantic coupling This rule is especially important for layout, selection, and high-frequency pattern edits.
Heavy paths must be intentional Undo, publish, or playback-reset behavior should happen only when the mutation actually warrants it. Value, Beat Lane, generated pattern, transport mutations Laggy interaction and "hope it doesn't glitch" feeling This is where lightweight pattern-library edits and heavier arrangement changes must stay distinct.

Chord Lab Contract Surfaces

Surface Role In Contract What It Consumes Notes
Processor mutation entry points Mutation origin and contract emitter Mutation intent and authoritative state changes This is where explicit invalidation types should ultimately originate.
Editor refresh domains Scoped UI consumers Only the invalidations relevant to each editor section Main editor responsiveness depends heavily on getting this boundary right.
Playback engine Time-domain and sound-domain consumer Transport, arrangement, Beat Lane, generated pattern, and preview invalidations that affect audio or cursors Playback authority and preview isolation should be defined by contract, not accident.
Pattern Generator session Local tool consumer Selection, session, generated pattern, and some transport/layout invalidations This is the clearest place where local session state must stay distinct from shared model state.

Design Tensions / Failure Modes

  • Weak contracts often show up as "the UI becomes correct only after some other stronger action," which means a mutation path is not fully declaring its consequences.
  • Generic refresh systems are easy to wire initially, but they quickly make Chord Lab feel expensive, fragile, and unpredictable under frequent edits.
  • Transport, Pattern Generator, and Beat Lane are the most contract-sensitive areas because they sit at the intersection of model truth, local session behavior, and playback authority.

Open Questions

  • Should Chord Lab define one shared invalidation type system across processor, editor, playback, and tools, or several small contract enums that map to each other?
  • Which existing processor mutation entry points already behave like strong contracts and can be formalized instead of reinvented?
  • How should undo/publish policy be represented in the contracts docs so it stays connected to responsiveness goals?
