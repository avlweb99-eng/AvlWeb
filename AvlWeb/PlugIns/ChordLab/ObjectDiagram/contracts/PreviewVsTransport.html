Preview vs Transport

Preview and transport are two different layers of behavior. Transport owns the main time source and playback authority; preview adds local or contextual audition behavior without becoming a competing owner of time. This contract matters because many of Chord Lab's brittle playback bugs come from preview behavior being routed through authority-owned paths or being allowed to act like a second transport.

Definition

Transport is the authoritative owner of playback time; preview is a local or contextual playback layer that can add or audition material under that authority. In Chord Lab, preview behavior includes Pattern Generator audition and similar local test playback, while transport is owned by the host or internal playback engine according to the authority contract. This contract protects the system from double playback, wrong cursor ownership, and preview changes accidentally resetting or corrupting main playback.

Layered behavior Transport owns time Preview stays scoped No silent overlap
Key distinction:
Preview can change what is heard locally, but it must not silently change who owns playback time or make the app behave as if a second transport is running.

What This Is Not

Is This Contract

Pattern Generator can audition a pattern while editing, but that audition must not leak chord notes into main playback or override host transport ownership.

Not This Contract

"Host transport always wins." That is the transport authority rule itself. This page governs how preview behaves under that authority.

Boundary Case

A local design-mode loop can feel transport-like to the user, but it still belongs to preview/session policy unless it actually owns the main playback timeline.

Preview vs Transport Diagram

Transport resolves the owner of time. Preview layers may add local playback behavior, but they must remain subordinate and scoped under that owner.

Examples

Placed-Beat Editing

Open Pattern Generator on a Beat Lane block. The cursor and audible behavior should follow the real placement in the arrangement, not a free-running local preview loop.

Design-Mode Audition

Open Pattern Generator in standalone design mode. The pattern may loop locally for audition, but this local loop is still a preview behavior layered under transport authority.

Preview Isolation Fix

Changing preview state should not inject chord notes into the main playback path. Preview events must stay isolated from normal arrangement playback unless explicitly intended.

Non-Examples

  • "Host playback disables internal controls" That belongs to the transport authority contract.
  • "Selecting a step updates the step inspector" That belongs to selection/session behavior, not the preview-layer contract.
  • "The Beat Lane block moved to bar 5" That is a Beat Lane mutation; preview only changes how the result may be auditioned.

Design Tensions / Failure Modes

  • Pattern Generator has repeatedly exposed preview-vs-transport confusion because local audition, placed-beat following, and arrangement playback all want different behavior from closely related code paths.
  • If preview state goes through the same heavy mutation and playback-reset path as arrangement truth, the tool feels brittle and laggy even when the conceptual edit is local.
  • Preview can feel transport-like to the user, which makes the boundary easy to blur unless the contract is explicit in both code and UI behavior.

Open Questions

  • Should preview-only updates eventually bypass more of the shared processor/playback state machinery entirely?
  • What is the cleanest long-term rule for Pattern Generator design-mode local looping when host transport is active versus inactive?
