Transport Authority

Transport authority defines which timeline owns playback at any given moment. In Chord Lab, this contract governs the relationship between host transport, internal transport, and any tool-local audition behavior layered on top of them. This contract matters because unclear authority leads directly to double playback, wrong cursors, disabled-state confusion, and the feeling that playback cannot be fully trusted.

Definition

Transport authority is the rule that determines which time source the app must follow for playback and transport-facing UI. In Chord Lab, host transport is authoritative whenever the host reports active playback; internal transport is valid only when the host is not playing. This contract protects the app from conflicting playback owners and makes transport behavior explicit instead of accidental.

Authority-first Host wins Internal is valid Preview is separate
Key distinction:
Transport authority is not the same thing as preview or audition. Preview can layer behavior on top of transport, but it must not silently become a second transport owner.

What This Is Not

Is This Contract

If the host starts playing, Chord Lab must follow host timing immediately and internal transport controls should no longer act as active transport controls.

Not This Contract

"The Pattern Generator can loop locally while I design a beat." That is a tool-local behavior policy, not transport authority by itself.

Boundary Case

Internal transport is first-class in Chord Lab, but it is still subordinate to host playback whenever the host is active. First-class does not mean co-equal authority.

Transport Authority Diagram

Transport authority takes transport state as input, resolves a single owner of time, and makes every transport-facing surface follow that owner.

Examples

Host Override

Internal transport is already running, then the host starts playback. Host immediately becomes authoritative and internal controls should visually reflect that they are no longer the active owner.

Internal Audition Workflow

The host is idle and the user presses play in Chord Lab. Internal transport is now the active authority and the Chord Roll/Beat Lane follow it normally.

Local Tool Under Authority

Pattern Generator design mode loops locally, but if host playback is active the tool still operates under host-owned playback authority rather than acting like a competing transport.

Non-Examples

  • "Preview notes should not leak into main playback" That is preview-layer isolation, not the authority rule itself.
  • "The cursor repainted at the right time" That is a consumer outcome of authority, not the contract.
  • "The Pattern Generator opened in design mode" That is a session mutation. It may imply local playback behavior, but it does not define transport ownership alone.

Transport Authority Rules

Rule Meaning Applies To Must Prevent Notes
Host transport is authoritative whenever host playback is active The host becomes the single owner of playback time as soon as it reports play-state. Playback engine, transport UI, cursors, arrangement-following views Internal transport continuing as a competing owner This is an explicit product requirement already agreed in the project.
Internal transport is first-class only while host is idle Chord Lab internal transport is a valid user workflow, but only when the host is not playing. Internal play/stop controls, internal audition flow, status text Internal transport being treated like a fallback-only hack or, conversely, like a co-equal host competitor “First-class” here means product support and UX clarity, not shared authority with the host.
Transport-facing UI must reflect authority immediately Buttons, labels, and enabled states must match the active authority as soon as it changes. Main transport bar, status labels, disabled states UI lying about which transport is currently in charge Internal play/stop should be disabled while host transport is active.
Tool-local audition must defer to authority Local playback behavior can exist, but it cannot silently become a second transport owner. Pattern Generator design mode, preview layers, local loops Double playback, wrong cursor ownership, competing play-state logic This rule is where authority and preview/session contracts intersect.

Design Tensions / Failure Modes

  • The playback engine already reflects host priority, but the UI has historically not made authority visible or enforce it strongly enough.
  • Internal transport can feel "second class" when it is only treated as a fallback in the product, even if the engine supports it properly.
  • Pattern Generator preview and design-mode looping expose authority confusion quickly because they sit at the edge of transport, preview, and local tool behavior.

Open Questions

  • Which exact transport controls should be disabled during host playback, and which should remain visible but passive?
  • Should host-active state be represented as a dedicated editor-wide status object so every transport-facing surface can consume it consistently?
  • How should future custom loop-region support interact with authority without making cursor ownership ambiguous?
