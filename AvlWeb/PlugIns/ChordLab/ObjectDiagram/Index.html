Mutations

Behavior Categories

Mutation types describe state changes, not just button clicks. These pages are now live and form the shared vocabulary for update contracts.

Selection MutationsChanges to the current edit target or focused object, global or tool-local.
Value MutationsDirect edits to chord values, progression values, or settings values.
Beat Lane MutationsPlace, move, delete, retarget, and kit-level changes in Beat Lane.
Generated Pattern MutationsCreate, update, or delete reusable beat patterns.
Transport MutationsHost/internal play state changes, transport authority changes, and cursor authority changes.
Session MutationsOpen/close tool sessions, switch Pattern Generator mode, change local preview targets.
Layout MutationsResize, tab changes, visibility changes, and non-musical layout state.

Important Distinction

Not every UI action is the same kind of mutation.

Selection mutations change what the app or tool is targeting for editing.
Value mutations change authoritative data.
Session mutations change tool context, such as opening the Pattern Generator in design mode.
Layout mutations should only affect bounds and visibility. They should not trigger hidden semantic refreshes.
How the set fits together
Selection chooses the target. Value changes stored data. Session changes tool context. Generated Pattern and Beat Lane mutations describe the two main beat domains. Transport defines playback authority. Layout keeps visual structure separate from semantic changes.

Update Contracts

Core Contract Pages

These pages define what each mutation invalidates and who must react. The contracts section is now started with a live overview page.

Update Contracts OverviewShared contract language, invalidation categories, and consumer model.
Transport AuthorityHost always wins, internal transport is first-class but subordinate, and transport-facing UI must reflect authority immediately.
Preview vs TransportHow local audition and preview layers stay subordinate to transport instead of acting like a second timeline owner.
Editor Refresh DomainsHow the main editor consumes invalidations in scoped domains instead of falling back to broad full refreshes.

Current Working Rule

The system should become correct immediately after the mutation and its declared contract consumers run.

No repair refreshes.
If one mutation only becomes visibly correct after a later stronger action, then the original update contract is incomplete.
No silent overlap.
Preview, audition, local tool playback, and transport authority must stay explicitly separated.
No generic full refresh by default.
Broad refreshes should be the exception, not the normal response to state change.
